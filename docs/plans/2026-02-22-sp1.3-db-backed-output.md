# SP1.3: DB-Backed Output

## Context

SP1.2 added the SQLite database layer (sqlx, DAO, migrations) alongside KDL
config and XDG paths. The existing binaries still use the old ScanState flow:
scan → in-memory HashMap → output. SP1.3 rewires both binaries to go through the
database: scan → DB → query DB → output. The old ScanState code is kept for now
(removed in SP1.4) but the binaries stop using it.

The hard gate: CSV file content and tree stdout must be byte-identical to the
current output. Both currently use ICU4X collation (Quaternary strength, Shifted
alternates) for sorting — the DB-backed versions must reproduce the same sort.

## Design changes from SP1.2

Three cross-cutting changes apply throughout SP1.3:

**u64 for all sizes**: Sizes are inherently unsigned scalar values. All new code
uses `u64` for sizes. The DAO's `FileInput.size` and `FileRecord.size` change
from `i64` to `u64`. SQLite stores integers as signed i64; the DAO casts at the
boundary (`as i64` when binding, `as u64` when reading). Date/time values
(`ctime`, `mtime`) remain `i64` — they are not sizes. Existing rkyv/ScanState
code is left as-is since it's going away in SP1.4.

**Capture everything, filter on output**: The new `scan_to_db()` captures ALL
paths — no `exclude` parameter. Filtering (ignore patterns, hidden files,
excluded directory names) moves entirely to the output layer (CSV writer, tree
renderer). This ensures accurate size calculations (directory + subtree sizes)
regardless of output filters. The scanner also stores directory entry sizes
(`metadata.size()`) for accurate du-like totals. New ADR documents this pattern.

**ICU4X collation for all output sorting**: Both CSV and tree use ICU4X
collation (this is already the case in the code). Update the existing ICU4X ADR
to reflect this, and fix the incorrect CLAUDE.md note that says CSV uses
byte-order sorting.

### 2. `scan_to_db` in scanner.rs

**File**: `etp-lib/src/scanner.rs`

Add alongside existing `scan()`:

```rust
pub async fn scan_to_db(
    root: &Path,
    pool: &SqlitePool,
    run_type: &str,
    verbose: bool,
) -> io::Result<(i64, ScanStats)>
```

No `exclude` parameter — captures everything.

Logic:

1. `dao::upsert_scan(pool, run_type, root_str)` → `scan_id`
2. WalkDir with NO filter_entry (capture all directories)
3. For each directory:
   - Compute relative path by stripping root prefix (empty string for root dir)
   - `dao::directory_mtime(pool, scan_id, relative_path)` for cache check
   - On cache miss: `scan_directory()` (reuse existing helper), then
     `dao::upsert_directory(pool, scan_id, relative_path, mtime, dir_size)` →
     `dir_id`, `dao::replace_files(pool, dir_id, files)`
   - On cache hit: record in `seen_paths` set, increment stats
   - Track `seen_paths` with **relative paths** (matching DB storage)
4. `dao::remove_stale_directories(pool, scan_id, &seen_paths)`
5. `dao::finish_scan(pool, scan_id)`
6. Return `(scan_id, stats)`

### 3. `write_csv_from_db` in csv_writer.rs

**File**: `etp-lib/src/csv_writer.rs`

Add alongside existing `write_csv()`:

```rust
pub async fn write_csv_from_db(
    pool: &SqlitePool,
    scan_id: i64,
    output: &Path,
    exclude: &[String],
) -> io::Result<()>
```

1. Call `dao::list_files(pool, scan_id)` → `Vec<FileRecord>`
2. Filter out files whose dir_path contains any excluded directory name
3. Group records by `dir_path` into `HashMap<String, Vec<FileRecord>>`
4. Sort dir keys with ICU4X collation (same settings as current code)
5. For each dir, sort its files by `filename` with same collation
6. Write CSV: `path,size,ctime,mtime` header, then
   `dir/filename,size,ctime,mtime`

### 4. `render_tree_from_db` in tree.rs

**File**: `etp-lib/src/tree.rs`

Add alongside existing `render_tree()`:

```rust
pub async fn render_tree_from_db(
    pool: &SqlitePool,
    scan_id: i64,
    root: &Path,
    patterns: &[Pattern],
    no_escape: bool,
    show_hidden: bool,
) -> (usize, usize)
```

1. Call `dao::list_directory_paths(pool, scan_id)` → all directory full paths
2. Call `dao::list_files(pool, scan_id)` → all file records
3. Build `children: BTreeMap<PathBuf, BTreeSet<String>>` from directory paths
   (same logic as current code — extract parent→child_name)
4. Build `files_by_dir: HashMap<String, Vec<String>>` grouping filenames by
   dir_path
5. Refactor `TreeContext` to work with either `&ScanState` or the map-based
   data, or create a parallel context struct
6. Render with same `render_dir` logic — patterns and show_hidden filter at
   render time, not at scan time

### 5. Update ops.rs with DB-backed functions

**File**: `etp-lib/src/ops.rs`

Add new async functions alongside existing ones:

- `run_scan_to_db(root, pool, run_type, verbose) -> i64` — calls
  `scanner::scan_to_db`, logs stats, returns scan_id
- `write_csv_from_db(pool, scan_id, output, exclude, verbose)` — calls
  `csv_writer::write_csv_from_db`, logs
- `render_tree_from_db(pool, scan_id, root, ignore, no_escape, show_hidden)` —
  calls `tree::render_tree_from_db`, prints summary

### 6. Rewire binary main.rs files

**Files**: `etp-csv/src/main.rs`, `etp-tree/src/main.rs`

Both binaries switch to `#[tokio::main(flavor = "current_thread")]`.

CLI changes:

- Keep `--state` as hidden/deprecated arg (accepted but ignored with warning)
- Add `--db <path>` (optional, defaults to `<directory>/.etp.db`)
- `--exclude` repurposed as output filter (applied in CSV writer and tree
  renderer, not during scanning). Both binaries share this filtering logic.
- Derive `run_type` from canonicalized directory path for now (SP1.4 uses
  config-based scan names)

Add `tokio` dependency to both binary Cargo.toml files:

```toml
tokio = { version = "1", features = ["rt", "macros"] }
```

**etp-csv flow**:

```
validate_directory → open_db → scan_to_db → write_csv_from_db (with exclude)
```

**etp-tree flow**:

```
validate_directory → open_db → scan_to_db → render_tree_from_db (with patterns)
```

### 7. Update trycmd tests

**File**: `etp-csv/tests/cmd/csv-verbose.stderr`

Update expected stderr to match new DB-backed messages. The state-related
messages ("no previous state", "saved state to") are gone.

All other trycmd tests should pass without changes (they check `stderr = ""` or
only stdout).

### 8. Integration test: old vs new path

**File**: `etp-lib/tests/db_output_parity.rs` (integration test)

Create a test that:

1. Creates a fixture directory with files and subdirectories
2. Scans with old path (`scan()` → `ScanState` → `write_csv()`)
3. Scans with new path (`scan_to_db()` → `write_csv_from_db()`)
4. Asserts CSV output is byte-identical
5. Similarly for tree: old path vs new path, assert stdout identical

This is the regression gate proving the DB path produces identical output.
